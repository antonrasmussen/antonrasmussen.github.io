Overview

This module will cover several types of general cyber attacks. This includes man in the middle attacks where an attacker will attempt to capture packets traveling through the network, denial of service attacks that attempt to prevent legitimate users from accessing a system, and phishing and spam attacks that attempt to obtain private information from users. We will also cover some defenses that can be used for email.

At the end of this module, students will be able to:

  • Explain the man in the middle attack.
  • Describe denial of service attacks and explain why they can be so damaging.
  • Distinguish between the smurf attack, ping of death, SYN flood, and UDP flood DoS attacks.
  • Explain why port scanning is an effective tool for attackers.
  • Describe how SPF and DKIM can be used to improve email security.
  • Explain how PGP provides confidentiality for email.

Attackers have lots of techniques that they can use to disrupt the legitimate use of the Internet. Knowing these techniques can help us to avoid the attacks and to develop appropriate defenses. Spam email and phishing techniques are a popular way for attackers to gain access to private information, often allowing them to perpetrate more attacks. Efforts to improve email security are essential in preventing some of these attacks.

Here are brief descriptions of four different DDoS attack methods.  (Note that these are DDoS attacks, so the attacker here is not just a single machine, but likely is a botnet.) 

Smurf attack – This attack uses ping (ICMP echo).  When a host receives an ICMP echo request, it will respond with an ICMP echo reply.  In this attack, the attackers issue an ICMP request to the broadcast address with a spoofed source IP address (the address of the target).  All hosts in the network will then flood the target with ICMP replies.

Ping of death – This attack also uses ping (ICMP echo).  In this case, the attacker will send a ping packet larger than the maximum IPv4 packet size (64KB), which could crash the target.  Most systems have fixed this bug, so this attack is not very effective today.

SYN flood – In TCP, when a SYN packet is received (1st packet in connection setup), the receiver sets aside resources needed to handle the TCP connection (buffer space, socket, etc.).  In this attack, the attacker will send many TCP SYN packets, but never complete the handshake (by responding to the receiver’s SYN/ACK).  This leaves many “half-open” connections on the receiver and can use up all of the receiver’s resources.  Legitimate connections will then not be able to be handled.

Optional: For more information on the SYN flood attack, see the "Focus On Security" highlight in the textbook in Section 3.5 (pg. 257 in 6th edition)

UDP flood – UDP has no flow control or congestion control.  A sender can send out UDP packets as fast as its network connection will allow.  Additionally, if a sender sends a UDP packet to a closed port, the receiver will respond with an ICMP Destination Unreachable message.  So, an attacker could send a large number of UDP packets to random ports, forcing the target to use its resources sending ICMP Destination Unreachable messages.

Although port scanning is not necessarily a DoS attack, it is a method that can be used to check for vulnerabilities.

Recall that the transport layer uses port numbers to help identify the process where a packet should be delivered.  Also recall that there are standard port numbers associated with well-known applications, for instance port 80 is HTTP, port 23 is the telnet protocol.  (See https://en.wikipedia.org/wiki/List_of_TCP_and_UDP_port_numbers for a list of well-known TCP and UDP port numbers.)

If an attacker knows that a certain application has a vulnerability, they may use a port scanner to find computers that are running that application and listening for connections on a particular port.  Once they find it, this computer may become the target for an attack.

The nmap tool (https://en.wikipedia.org/wiki/Nmap, https://nmap.org/) is a well-known port scanner, originally developed in 1997.  Nmap is also used by security professionals to perform security audits on their systems.

One of the nice things that nmap.org includes is a list of security tools.  See http://sectools.org/ and look at some of the different tools that are available.

Phishing

Phishing attacks are used to obtain sensitive personal information, such as social security numbers, credit card numbers, passwords, by masquerading as a legitimate website or email.  Often phishing emails and websites have mis-spellings or grammatical errors that give them away, but some are quite sophisticated and realistically copy the look and feel of a legitimate website.

Before we talk about specific techniques, let’s look at the basic approach of securing email, as described in the textbook in Section 8.5.1.

Confidentiality

In Figure 8.19, Alice uses a symmetric session key KS to send a secret email to Bob.  This is demonstrating confidentiality. Alice uses a symmetric key for encrypting the message because symmetric encryption is much faster and more efficient than asymmetric (public-key) encryption.

  • Alice picks a random symmetric session key KS
  • Alice encrypts the message m with the secret session key KS 
    • KS (m)
  • Alice encrypts the secret session key KS with Bob’s public key
    • KB+(KS)
  • Alice concatenates the encrypted message and the encrypted symmetric key to form a package
    • KS (m) + KB+ (KS)
  • Alice sends the package to Bob

Here, Alice is using the fast method to encrypt a potentially long message and the slower method to encrypt the shared key (which presumably is shorter than the email message).

At the other end, Bob will decrypt the message.

  • Bob de-concatenates the package into KB+ (KS) and KS (m)
  • Bob applies his private key to the encrypted symmetric key
    • KB- (KB+ (KS)) = KS
  • Bob applies the symmetric key to the encrypted message
    • KS (KS (m)) = m

Authentication and Message Integrity

Now let's look at the scenario if Alice only wants to send an authenticated message to Bob.  She doesn’t care if everyone in the world reads the message, but she wants to make sure that Bob knows it’s from Alice and that it hasn’t been tampered with.

This is the scenario shown in Figure 8.20, demonstrating authentication and message integrity.

  • Alice applies a hash function H to her message m to generate the message digest
    • H(m)
  • Alice then signs this result with her private key KA- to create a signed message digest
    • KA- (H(m))
  • Alice concatenates the original unencrypted message m with the signed message digest to form a package
    • m + KA- (H(m))
  • Alice sends the package to Bob

At the other end, Bob will verify that the message is from Alice and hasn’t been modified.

  • Bob de-concatenates the package into m and KA- (H(m))
  • Bob applies Alice’s public key to the signed message digest
    • KA+ (KA- (H(m))) = H(m)
  • Bob computes the hash of the message
    • H(m)
  • Bob then compares the results
    • H(m) == H(m)

Note that here Alice and Bob must agree on the hash function so that they will get the same answer.

Confidentiality, Authentication, and Integrity

Now, let's look at putting these all together.  We will combine the procedures described above.  Alice’s procedure is shown in Figure 8.21.

  • Alice creates the preliminary package, which is the signed message digest concatenated with the unencrypted message m (same as in the authentication and integrity procedure)
    • m + KA- (H(m))
  • Alice treats this package as a message to be encrypted.  Alice encrypts the package with the session key KS
    • KS (m + KA- (H(m)))
  • Alice encrypts the secret session key KS with Bob’s public key
    • KB+ (KS)
  • Alice concatenates these to form the package
    • KB+ ( KS) + KS (m + KA- (H(m)))
  • Alice sends the package to Bob

Note that here, Alice uses public key cryptography twice (with her private key and with Bob’s public key).

This is the basic procedure for providing secure email.  The remaining issue is in distributing and validating the public keys.  Alice has to trust that Bob’s public key is really his, and Bob has to trust that Alice’s public key is really hers. 

Secure Email (PGP and S/MIME)

Source: http://imgs.xkcd.com/comics/pgp.png

As we have seen, common techniques for securing email are based on the public key encryption techniques that we have discussed earlier. Pretty Good Privacy (PGP) provides confidentiality by combining both symmetric and public key encryption, and can also use digital signatures for authentication.  Secure/Multipurpose Internet Mail Extensions (S/MIME) provides confidentiality, along with authentication, integrity, and non-repudiation.

Here are the main differences (at least for us, looking at this at a high level):

PGP S/MIME
Signature is sent in plaintext Signature/certificate is sent in binary format (.p7s extension), mail client must know how to read it
Distributed web of trust model Centralized trust model – trust the CA
Public key sent separately from the signature Public certificate (and key) sent with each signature
User must generate and manage keys, signature (no certificate from a CA) Get a certificate from a CA (some are free)

Email Validation (DKIM and SPF)

Here we'll look a bit at email sender authentication.  This helps legitimate senders prove that the email they are sending isn’t spam.  Spammers often use forged IP addresses, which allows them to hide and keep their actual locations secret. 

For a look back at spam, read this 2006 article from security expert Bruce Schneier, “Why Spam Won’t Go Away”, https://www.schneier.com/essays/archives/2006/12/why_spam_wont_go_awa.html

DKIM

DomainKeys Identified Mail (DKIM) allows a domain (or email provider) to sign an email message and claim responsibility for it.  The receiving email server can then verify the signature to ensure the validity of the sender. DKIM is transparent to the email user.

Here's an example of how DKIM might be deployed.

  • A message is generated by a user and sent to the user’s local mail server.
  • The content of the email message, plus some of the email headers, is signed with the email provider’s private key.
  • The signed message is then sent through the Internet to the recipient’s local mail server.
  • The local mail server then retrieves the public key (using DNS) and verifies the signature.
  • If the signature is verified, then the message is put in the recipient’s mailbox.

SPF

The goal of Sender Policy Framework (SPF) is to prevent spammers from sending email with forged return IP addresses. In SPF, a domain would publish additional DNS resource records that identify the machines that are authorized to send email from that domain.  A receiving mail server could check the DNS record to verify if the address in the MAIL-FROM header is valid.

SPF is similar to DKIM, but SPF only validates the message headers (SMTP headers), not the message contents. 

The advantage to SPF is that the illegitimate email can be detected before the entire message is downloaded.
